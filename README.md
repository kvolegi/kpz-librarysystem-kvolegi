## Опис
Library System — це REST API система обліку книг бібліотеки. Вона дозволяє зручно керувати каталогом книг, здійснювати пошук за автором, назвою та жанром, а також бронювати книги для читачів з відстеженням термінів повернення.

## Технології
- **Мова програмування:** Python 3.12
- **Фреймворк:** FastAPI
- **База даних:** PostgreSQL 16
- **ORM:** SQLAlchemy
- **Міграції:** Alembic
- **Документація API:** Swagger UI (вбудована у FastAPI)
- **Контейнеризація:** Docker & Docker Compose

## Як запустити

### Передумови
- Python 3.12+
- Docker & Docker Compose (для запуску PostgreSQL)
- Git

### 1. Клонування репозиторію

```bash
git clone https://github.com/kvolegi/kpz-librarysystem-kvolegi.git
cd kpz-librarysystem-kvolegi
```

### 2. Створення та активація віртуального середовища

```bash
# Створення venv
python3 -m venv venv

# Активація (macOS / Linux)
source venv/bin/activate

# Активація (Windows)
venv\Scripts\activate
```

### 3. Встановлення залежностей

```bash
pip install -r requirements.txt
```

### 4. Налаштування змінних середовища

```bash
# Скопіюйте приклад і заповніть своїми значеннями
cp .env.example .env
```

Відкрийте `.env` та вкажіть параметри підключення до бази даних:
```
DATABASE_URL=postgresql://user:password@localhost:5432/library_db
SECRET_KEY=your-secret-key-here
```

### 5. Запуск PostgreSQL через Docker

```bash
docker-compose up -d
```

### 6. Застосування міграцій бази даних

```bash
alembic upgrade head
```

### 7. Запуск сервера

```bash
uvicorn app.main:app --reload
```

API буде доступне за адресою: **http://localhost:8000**  
Swagger документація: **http://localhost:8000/docs**

## Відповіді на запитання для самоперевірки

**1. Яка різниця між `git add` та `git commit`?**

`git add` переміщує змінені файли до *staging area* (зони підготовки) — це немов ви складаєте речі у валізу перед поїздкою. `git commit` робить знімок усього, що є у staging area, і зберігає його в історії репозиторію з повідомленням — це момент, коли валіза застібнута і готова. Без `git add` файл не потрапить до коміту, навіть якщо він змінений.

**2. Навіщо потрібен `.gitignore`?**

Щоб не зберігати в репозиторії файли, які: шкодять безпеці або не потрібні іншим розробникам. Три приклади файлів, які не варто зберігати:
- `.env` — містить паролі, API-ключі та секрети
- `venv/` — папка з встановленими Python-пакетами (кожен встановлює їх сам через `requirements.txt`)
- `__pycache__/` — скомпільовані Python-файли, генеруються автоматично

**3. Чому GitHub Copilot не можна використовувати без перевірки?**

Copilot — статистична модель, яка передбачає "найімовірніший" код, але не розуміє вашу бізнес-логіку. Приклад помилки: якщо попросити Copilot написати SQL-запит, він може згенерувати код вразливий до SQL-ін'єкцій, бо такі патерни часто зустрічались у навчальних даних. Або він може використати застарілу версію API бібліотеки. Завжди перевіряйте код, особливо у питаннях безпеки та роботи з даними.

**4. Яке повідомлення до коміту правильніше?**

`fix: виправити помилку розрахунку ПДВ у функції calculateTotal` — значно краще, ніж `fix bug`. Причини:
- **Стандарт** Conventional Commits дозволяє автоматично генерувати CHANGELOG та версії пакету

## Ваша Практична Робота: Аналіз та Вимоги

### 1. Ролі в системі
- **Гість (Guest)**: не авторизований відвідувач, який може переглядати список книг та здійснювати пошук у каталозі бази бібліотеки.
- **Читач (Reader)**: авторизований користувач, має свій акаунт, може бронювати виділені книги на певний термін та переглядати історію своїх бронювань.
- **Бібліотекар (Librarian / Admin)**: управляє книгами (додає, редагує, видаляє), а також управляє обліковими записами користувачів та актуалізує статуси бронювання.

### 2. Пріоритизація за MoSCoW (Таблиця)

| Категорія | Кількість | User Stories (заголовки) |
| --- | --- | --- |
| **Must Have** | 5 | 1. Реєстрація нового читача<br>2. Перегляд каталогу книг<br>3. Пошук книг за параметрами<br>4. Бронювання книги<br>5. Додавання нової книги (Бібліотекар) |
| **Should Have** | 3 | 6. Перегляд історії бронювань (Читач)<br>7. Редагування інформації про книгу (Бібліотекар)<br>8. Зміна статусу бронювання (Бібліотекар) |
| **Could Have** | 2 | 9. Сповіщення про термін повернення книги<br>10. Залишення відгуку до книги |
| **Won't Have** | 2 | 11. Онлайн читання (завантаження PDF)<br>12. Чат між читачами |

### 3. User Stories

#### [User Story] 1. Реєстрація нового читача
**Як** гість, **я хочу** зареєструватися в системі через email та пароль, **щоб** отримати можливість бронювати книги та переглядати свою історію.
*Acceptance Criteria:*
- [ ] GIVEN: сторінка реєстрації; WHEN: введено коректні дані; THEN: акаунт успішно створено, користувач перенаправлений у кабінет.
- [ ] GIVEN: форма; WHEN: email вже існує; THEN: помилка "Цей email вже використовується".
- [ ] GIVEN: форма; WHEN: пароль містить менше 8 символів; THEN: підказка про недостатню довжину.

#### [User Story] 2. Перегляд каталогу книг
**Як** гість, **я хочу** переглядати повний список книг у бібліотеці, **щоб** обрати цікаву для себе літературу.
*Acceptance Criteria:*
- [ ] GIVEN: головна сторінка; WHEN: користувач заходить на сторінку; THEN: відображається список доступних книг з обкладинками.
- [ ] GIVEN: каталог; WHEN: книг більше ніж 20; THEN: працює пагінація на сторінці.
- [ ] GIVEN: каталог; WHEN: книг немає; THEN: виводиться повідомлення "Каталог порожній".

#### [User Story] 3. Пошук книг за параметрами
**Як** гість, **я хочу** шукати книгу за назвою або автором, **щоб** швидко знаходити потрібні матеріали.
*Acceptance Criteria:*
- [ ] GIVEN: поле пошуку; WHEN: введено точну назву книги; THEN: відображається картка цієї книги.
- [ ] GIVEN: поле пошуку; WHEN: введено частину прізвища автора; THEN: відображаються всі книги цього автора.
- [ ] GIVEN: поле пошуку; WHEN: збігів не знайдено; THEN: повідомлення "За вашим запитом нічого не знайдено".

#### [User Story] 4. Бронювання книги
**Як** читач, **я хочу** забронювати певну книгу, **щоб** зарезервувати її для себе та пізніше забрати в бібліотеці.
*Acceptance Criteria:*
- [ ] GIVEN: сторінка книги в наявності (авторизований); WHEN: натиснуто "Забронювати"; THEN: створюється запис бронювання, книга резервується.
- [ ] GIVEN: сторінка книги; WHEN: не авторизований; THEN: кнопка веде на логін з текстом "Спочатку увійдіть".
- [ ] GIVEN: сторінка книги; WHEN: всі примірники заброньовані; THEN: кнопка неактивна.

#### [User Story] 5. Додавання нової книги
**Як** бібліотекар, **я хочу** мати можливість додавати нові книги до системи, **щоб** оновлювати каталог.
*Acceptance Criteria:*
- [ ] GIVEN: адмін-панель; WHEN: заповнено назву, автора, жанр та кількість; THEN: книга успішно додається.
- [ ] GIVEN: форма; WHEN: пропущено обов’язкове поле; THEN: поле підсвічується червоним, не зберігається.
- [ ] GIVEN: форма; WHEN: введено кількість менше 0; THEN: помилка валідації.

#### [User Story] 6. Перегляд історії бронювань
**Як** читач, **я хочу** переглядати історію своїх бронювань, **щоб** пам'ятати, які книги в мене на руках або які читав.
*Acceptance Criteria:*
- [ ] GIVEN: кабінет; WHEN: вкладка "Історія"; THEN: таблиця всіх бронювань.
- [ ] GIVEN: історія; WHEN: книга зараз 'на руках'; THEN: статус "Активно", показано кінцеву дату.
- [ ] GIVEN: історія; WHEN: щойно зареєструвався; THEN: "У вас ще немає історії бронювань".

#### [User Story] 7. Редагування інформації про книгу
**Як** бібліотекар, **я хочу** редагувати існуючі книги, **щоб** виправляти помилки або оновлювати залишок.
*Acceptance Criteria:*
- [ ] GIVEN: адмін-панель; WHEN: кнопка "Редагувати"; THEN: форма із заповненими даними.
- [ ] GIVEN: форма редагування; WHEN: змінено кількість примірників; THEN: оновлення всюди.
- [ ] GIVEN: дії над книгою; WHEN: редагує звичайний користувач (API); THEN: 403 Forbidden.

#### [User Story] 8. Зміна статусу бронювання
**Як** бібліотекар, **я хочу** змінювати статус бронювання ('Видана' / 'Повернута'), **щоб** вести облік використання.
*Acceptance Criteria:*
- [ ] GIVEN: активні бронювання; WHEN: натиснуто "Видати"; THEN: статус змінюється на "Видана".
- [ ] GIVEN: видані книги; WHEN: натиснуто "Повернути"; THEN: статус "Повернута", залишок збільшується.
- [ ] GIVEN: бронювання; WHEN: статус фінальний; THEN: запис зникає з активних черг.

#### [User Story] 9. Сповіщення про термін повернення
**Як** читач, **я хочу** отримувати повідомлення про наближення кінцевого терміну, **щоб** уникнути штрафів.
*Acceptance Criteria:*
- [ ] GIVEN: бронювання; WHEN: 1 день до закінчення; THEN: надсилається email.
- [ ] GIVEN: бронювання; WHEN: прострочено; THEN: статус підсвічується "Прострочено".
- [ ] GIVEN: бронювання; WHEN: повернуто вчасно; THEN: сповіщення скасовуються.

#### [User Story] 10. Залишення відгуку до книги
**Як** читач, **я хочу** залишати відгуки та оцінку книзі, **щоб** порадити її іншим читачам.
*Acceptance Criteria:*
- [ ] GIVEN: успішно повернута книга; WHEN: читач зберігає текст відгуку; THEN: публікується під книгою.
- [ ] GIVEN: сторінка книги; WHEN: читач її не брав; THEN: форма заблокована.
- [ ] GIVEN: сторінка книги; WHEN: більше 2 відгуків; THEN: відображається середній рейтинг.
## Ролі користувачів
- **Гість (Guest)**: неавторизований відвідувач, може переглядати каталог та шукати книги.
- **Читач (Reader)**: авторизований користувач, може бронювати книги та переглядати історію.
- **Бібліотекар (Librarian)**: управляє книгами, користувачами та статусами бронювань.
