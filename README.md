## Опис
Library System — це REST API система обліку книг бібліотеки. Вона дозволяє зручно керувати каталогом книг, здійснювати пошук за автором, назвою та жанром, а також бронювати книги для читачів з відстеженням термінів повернення.

## Технології
- **Мова програмування:** Python 3.12
- **Фреймворк:** FastAPI
- **База даних:** PostgreSQL 16
- **ORM:** SQLAlchemy
- **Міграції:** Alembic
- **Документація API:** Swagger UI (вбудована у FastAPI)
- **Контейнеризація:** Docker & Docker Compose

## Як запустити

### Передумови
- Python 3.12+
- Docker & Docker Compose (для запуску PostgreSQL)
- Git

### 1. Клонування репозиторію

```bash
git clone https://github.com/kvolegi/kpz-librarysystem-kvolegi.git
cd kpz-librarysystem-kvolegi
```

### 2. Створення та активація віртуального середовища

```bash
# Створення venv
python3 -m venv venv

# Активація (macOS / Linux)
source venv/bin/activate

# Активація (Windows)
venv\Scripts\activate
```

### 3. Встановлення залежностей

```bash
pip install -r requirements.txt
```

### 4. Налаштування змінних середовища

```bash
# Скопіюйте приклад і заповніть своїми значеннями
cp .env.example .env
```

Відкрийте `.env` та вкажіть параметри підключення до бази даних:
```
DATABASE_URL=postgresql://user:password@localhost:5432/library_db
SECRET_KEY=your-secret-key-here
```

### 5. Запуск PostgreSQL через Docker

```bash
docker-compose up -d
```

### 6. Застосування міграцій бази даних

```bash
alembic upgrade head
```

### 7. Запуск сервера

```bash
uvicorn app.main:app --reload
```

API буде доступне за адресою: **http://localhost:8000**  
Swagger документація: **http://localhost:8000/docs**

## Відповіді на запитання для самоперевірки

**1. Яка різниця між `git add` та `git commit`?**

`git add` переміщує змінені файли до *staging area* (зони підготовки) — це немов ви складаєте речі у валізу перед поїздкою. `git commit` робить знімок усього, що є у staging area, і зберігає його в історії репозиторію з повідомленням — це момент, коли валіза застібнута і готова. Без `git add` файл не потрапить до коміту, навіть якщо він змінений.

**2. Навіщо потрібен `.gitignore`?**

Щоб не зберігати в репозиторії файли, які: шкодять безпеці або не потрібні іншим розробникам. Три приклади файлів, які не варто зберігати:
- `.env` — містить паролі, API-ключі та секрети
- `venv/` — папка з встановленими Python-пакетами (кожен встановлює їх сам через `requirements.txt`)
- `__pycache__/` — скомпільовані Python-файли, генеруються автоматично

**3. Чому GitHub Copilot не можна використовувати без перевірки?**

Copilot — статистична модель, яка передбачає "найімовірніший" код, але не розуміє вашу бізнес-логіку. Приклад помилки: якщо попросити Copilot написати SQL-запит, він може згенерувати код вразливий до SQL-ін'єкцій, бо такі патерни часто зустрічались у навчальних даних. Або він може використати застарілу версію API бібліотеки. Завжди перевіряйте код, особливо у питаннях безпеки та роботи з даними.

**4. Яке повідомлення до коміту правильніше?**

`fix: виправити помилку розрахунку ПДВ у функції calculateTotal` — значно краще, ніж `fix bug`. Причини:
- **Стандарт** Conventional Commits дозволяє автоматично генерувати CHANGELOG та версії пакету

## Лабораторна робота 2: Специфікація вимог

### 1. Ролі користувачів
- **Гість (Guest)**: неавторизований відвідувач, може переглядати каталог та шукати книги.
- **Читач (Reader)**: авторизований користувач, може бронювати книги та переглядати історію.
- **Бібліотекар (Librarian)**: управляє книгами, користувачами та статусами бронювань.

### 2. Пріоритизація за MoSCoW

| Категорія | Кількість | User Stories |
| --- | --- | --- |
| **Must Have** | 5 | Реєстрація, Авторизація, Додавання книг, Пошук, Бронювання |
| **Should Have** | 3 | Історія читання, Редагування картки, Скасування броні |
| **Could Have** | 3 | Темна тема, Експорт у CSV, Відгуки |
| **Won't Have** | 3 | Мобільний додаток, Календар, PDF-читання |

### 3. User Stories та Acceptance Criteria

#### 3.1. Must Have (обов'язкова функціональність)

**US-01: Реєстрація нового читача**
| Acceptance Criteria | Опис перевірки |
| :--- | :--- |
| **AC1: успішна реєстрація** | GIVEN: форма реєстрації \| WHEN: введено email у форматі `name@domain.com` та пароль 8+ символів \| THEN: акаунт створено, редирект на dashboard |
| **AC2: дублікат email** | GIVEN: форма реєстрації \| WHEN: введено вже зареєстрований email \| THEN: повідомлення «Email вже використовується» |
| **AC3: слабкий пароль** | GIVEN: поле пароль \| WHEN: пароль < 8 символів \| THEN: inline-підказка «Мінімум 8 символів», кнопка неактивна |

**US-02: Авторизація користувача (Login)**
| Acceptance Criteria | Опис перевірки |
| :--- | :--- |
| **AC1: успішний вхід** | GIVEN: форма входу \| WHEN: введено коректні дані (email/пароль) \| THEN: JWT токен у відповіді, редирект на /dashboard |

| **AC2: порожня назва** | GIVEN: форма книги \| WHEN: заголовок (Title) порожній \| THEN: кнопка 'Зберегти' неактивна |

#### 3.2. Should Have (важлива функціональність)

**US-06 — Перегляд історії читання (Reader)**
**User Story:** Як читач, я хочу переглядати список усіх книг, які я брав раніше, щоб контролювати свою активність та пам'ятати прочитане.
* WHEN: обрано вкладку «Історія» THEN: відображається перелік усіх повернутих книг з датами видачі та повернення
* WHEN: натиснуто на назву книги THEN: перехід на сторінку книги з детальним описом
* WHEN: користувач не брав жодної книги THEN: повернути порожній список з повідомленням «Ви ще не брали книг»

**US-07 — Редагування картки книги (Librarian)**
**User Story:** Як бібліотекар, я хочу змінювати дані про книги, щоб виправляти помилки в описі або оновлювати статус видань.
* WHEN: змінено текст опису та натиснуто «Зберегти» THEN: дані оновлено в БД, статус успіху «Зміни збережено»
* WHEN: введено нове число у поле «Кількість примірників» THEN: книга автоматично стає доступною для відповідної кількості бронювань
* WHEN: видалено назву (поле порожнє) THEN: інлайн-помилка «Назва не може бути порожньою», збереження не відбувається

**US-08 — Скасування бронювання (Reader)**
**User Story:** Як читач, я хочу мати можливість скасувати своє бронювання онлайн, якщо мої плани змінилися.
* WHEN: натиснуто кнопку «Скасувати» THEN: статус книги змінюється на 'available', вона з'являється в пошуку для інших
* WHEN: з'являється вікно підтвердження і користувач натискає «Так» THEN: запис видаляється з активних бронювань
